# 서버 구조 정리 — 왜 서버가 2개 필요한가?

> 작성일: 2026-04-03
> 
> 작성자: 장성준
> 


---

## 1. 핵심 요약

Hearo 프로젝트는 **2개의 서버**가 필요합니다.

| | RPi 로컬 서버 | Firebase 클라우드 |
|--|-------------|-----------------|
| **위치** | 집 안 라즈베리파이 | 구글 클라우드 |
| **역할** | AI 추론 + ESP32 통신 | 인증 + 스마트폰 푸시 알림 |
| **인터넷 필요** | 불필요 | 필요 |
| **비용** | 0원 (자체 하드웨어) | 무료 (Firebase 무료 티어) |

---

## 2. 왜 2개인가?

### RPi 로컬 서버 (서버 1)

ESP32가 수집한 소리를 받아서 AI로 판단하고, 다시 ESP32에 결과를 전달하는 **중앙 허브** 역할입니다.

하는 일:
- MQTT 브로커: ESP32들과 데이터를 주고받는 통신 허브
- YAMNet AI 추론: 받은 소리가 뭔지 판단 (화재경보, 초인종 등)
- ESP32에 결과 전달: 모든 ESP32에 보내서 LED/진동 작동

이 서버가 없으면 ESP32들은 소리만 듣고 아무것도 할 수 없습니다.

### Firebase 클라우드 (서버 2)

사용자가 **외출 중일 때** 스마트폰으로 알림을 보내기 위해 필요합니다.

하는 일:
- Firebase Auth: 회원가입/로그인
- Firebase Cloud Messaging (FCM): 스마트폰 푸시 알림 전달
- Firestore: 사용자-기기 매핑 (누구 집 RPi인지)

스마트폰 푸시 알림은 Google/Apple 서버를 반드시 거쳐야 하기 때문에, RPi가 직접 스마트폰에 알림을 보낼 수 없습니다. Firebase가 이 **배달부** 역할을 합니다.

---

## 3. 전체 동작 흐름

```
[1] ESP32 (각 방에서 소리 수집)
     │
     ▼ MQTT 전송 (WiFi, 로컬)
[2] RPi 서버 (소리 수신 → AI 추론 → "화재경보!")
     │
     ├──[3a] MQTT → 모든 ESP32 → LED/진동 (로컬, 즉시, ~10ms)
     │
     └──[3b] Firebase FCM → 스마트폰 푸시 알림 (클라우드, 1~3초)
```

### 인터넷 끊겼을 때

```
ESP32 → RPi (AI 추론) → ESP32 LED/진동  ← 정상 동작
                       ✗ Firebase 연결 불가 → 푸시 알림 못 보냄
```

집 안의 LED/진동 알림은 인터넷 없이도 정상 동작합니다.

---

## 4. 왜 ESP32로 다시 보내야 하는가?

최종 엔드포인트가 스마트폰 알림이지만, ESP32 LED/진동도 필수입니다.

| 상황 | 스마트폰만 | ESP32 LED/진동 |
|------|----------|---------------|
| 핸드폰 다른 방에 둠 | 못 봄 | 각 방에서 알림 |
| 핸드폰 무음 모드 | 놓침 | LED 빛으로 인지 |
| 자고 있을 때 | 못 봄 | 진동+LED로 깨움 |
| 인터넷 끊김 | 푸시 불가 | 정상 동작 |

청각장애인은 소리를 들을 수 없기 때문에, **시각(LED) + 촉각(진동)**으로 전달하는 것이 핵심 가치입니다.
