## 1. 문제 인식

기존 계획은 라즈베리파이를 각 방마다 1대씩 설치하는 구조였음.

- RPi 1대당 약 20만원 (본체 + 마이크 + LED + 진동 + 부품)
- 3개 방 설치 시: **약 60만원**
- 프로젝트 목표가 "저렴한 비용으로 청각장애인 가정에 보급"인데, 비용이 너무 높음

## 2. 변경안: 허브-노드 구조 (RPi 1대 + ESP32 N대)

### 핵심 아이디어

라즈베리파이는 AI 두뇌 역할만 하고, 각 방에는 저렴한 ESP32를 배치한다.

### 구조도

```
[주방 ESP32 + 마이크 + LED] ──WiFi──┐
[거실 ESP32 + 마이크 + LED] ──WiFi──┤──→ [RPi 허브 1대: AI 추론]
[침실 ESP32 + 마이크 + LED] ──WiFi──┘         │
         ↑                                    │
         └──── MQTT로 결과 수신 ←──────────────┘
       (각 ESP32가 LED/진동 알림)
```

### 각 기기의 역할

| 기기 | 위치 | 역할 | 코드 언어 |
|------|------|------|-----------|
| **RPi 허브** (1대) | 집 안 아무 곳 (눈에 안 보여도 됨) | 소리 데이터 수신 → YAMNet AI 추론 → 결과 판단 → 앱/ESP32에 전달 | Python |
| **ESP32 노드** (N대) | 각 방 벽면에 부착 | 마이크로 소리 수집 → RPi에 전송 → 결과 받아서 LED/진동 알림 | C/C++ (Arduino) |

### 데이터 흐름

```
1. [ESP32] 마이크로 소리 수집
       ↓ WiFi 전송
2. [RPi 허브] 소리 데이터 수신 → YAMNet AI로 분류
       ↓
3. [RPi 허브] 분류 결과를 두 곳에 동시 전달
       ├──→ [모든 ESP32] "화재경보" → LED 빨간색 + 진동
       └──→ [앱/Firebase] 푸시 알림 전송
```

### 왜 ESP32가 직접 앱에 보내지 않는가

- ESP32는 소리를 **듣기만** 할 수 있고, AI 모델을 돌릴 성능이 안 됨
- 직접 앱에 보내면 "소리가 났다"만 전달 가능, "무슨 소리인지"는 판단 불가
- 반드시 RPi를 거쳐야 소리 종류를 분류할 수 있음

## 4. 장점 및 고려사항

### 장점

- 비용 약 45% 절감 (46.5만 → 25.4만)
- 방 확장 시 ESP32만 추가하면 되므로 대당 ~3만원
- 온디바이스 추론 (인터넷 끊겨도 RPi에서 AI 동작)
- 프로젝트 취지("저렴한 가격의 보급형 시스템")에 부합

### 고려사항

- ESP32용 코드를 별도로 개발해야 함 (C/C++, Arduino IDE)
  - 단, ESP32가 하는 일이 단순해서(소리 녹음 + WiFi 전송 + LED/진동) 코드 난이도 낮음
- WiFi 의존: 같은 공유기 내에서 동작 (지연 ~10ms 수준으로 문제 없음)
- RPi 허브가 고장나면 전체 시스템 중단 → 안정성 확보 필요
