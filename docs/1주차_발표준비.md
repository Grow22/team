## 1. 과제 목적 및 필요성 (Why)

### 1-0. 기획 의도 — "소프트웨어조차 활용하기 힘든 약자를 위해"

이 프로젝트의 출발점은 하나의 질문이었습니다.

> **"ICT 기술의 혜택을 가장 받기 어려운 사람은 누구인가?"**

사회적 취약 계층의 범위는 넓습니다. 노인, 장애인, 다문화 가정, 저소득층 등 다양한 대상이 있습니다. 하지만 이 중 상당수는 스마트폰 앱이나 웹 서비스를 통해 어느 정도 도움을 받을 수 있습니다. 화면을 보고, 소리를 듣고, 터치할 수 있는 사람이라면 기존의 소프트웨어적 솔루션으로도 접근이 가능하기 때문입니다.

그러나 **청각장애인은 다릅니다.** 아무리 좋은 앱이 설치되어 있어도, 소리 기반의 알림을 들을 수 없습니다. 아기가 울어도, 초인종이 울려도, 화재경보가 울려도 — 소리라는 채널 자체가 차단되어 있는 사람에게 소프트웨어만으로는 근본적인 해결이 불가능합니다.

특히 경제적으로 여유롭지 않은 환경에서 살아가는 청각장애인은 고가의 보조 장비를 갖추기 어렵고, 가정 내에서 도움을 줄 사람이 없는 경우도 많습니다. 이들은 기술의 사각지대 중에서도 가장 깊은 곳에 놓여 있습니다.

이 프로젝트는 바로 그 지점에서 시작되었습니다.

> **"소프트웨어적 기능에 하드웨어적 보완을 더해, 소리를 들을 수 없는 사람에게 다른 감각(시각, 촉각)으로 위험을 전달하자."**

소프트웨어(AI 오디오 분류)가 소리를 "이해"하고, 하드웨어(LED 알림, 진동 모터)가 그 결과를 시각과 촉각으로 "전달"하는 구조입니다. 이 SW+HW 결합이 이 프로젝트의 핵심 기획 의도이자, 순수 소프트웨어 솔루션과의 근본적 차별점입니다.

### 1-0-0. 주제 선정의 3가지 기준

저희 조가 이 주제를 최종 선정하기까지, 다양한 사회공헌 주제를 검토했습니다. 그 과정에서 세 가지 기준을 세웠고, 이 주제는 세 기준을 모두 충족한 유일한 후보였습니다.

**기준 1. 사회적 당위성 — "정말 도움이 필요한 대상인가?"**

사회적 약자 계층 중에서도 기존 소프트웨어만으로는 근본적 해결이 불가능한 대상을 찾고자 했습니다. 청각장애인은 소리라는 채널 자체가 차단되어 있어, 아무리 좋은 앱이 있어도 화재경보나 초인종 소리를 들을 수 없습니다. SW에 HW를 결합해야만 비로소 도움을 줄 수 있는 대상입니다.

**기준 2. 시장 공백 — "이미 잘 만들어진 솔루션이 있는가?"**

기존에 보편화된 서비스가 있는 영역에 학부생 팀이 뛰어드는 건 비효율적입니다. 저희는 기존 시장을 철저히 조사한 결과, **한국에 AI 기반 가정 환경음 분류 전용 IoT 시스템이 없다**는 것을 확인했습니다. 구글 Sound Notifications는 스마트폰 단일 마이크 한계가 있고, 해외 서비스(Earzz 등)는 한국에 진출하지 않았으며, 한국 가정 환경음(아파트 인터폰, 밥솥, 세탁기 등)에 특화된 솔루션은 존재하지 않습니다. 이 시장 공백이 학부 프로젝트로서 차별화된 기여를 할 수 있는 여지를 만들어줍니다.

**기준 3. 구현 가능성 — "5명이 7개월 안에 완성할 수 있는가?"**

아무리 좋은 아이디어도 구현하지 못하면 의미가 없습니다. 이 프로젝트의 핵심 기술인 AI 오디오 분류(YAMNet 전이학습)와 IoT 디바이스(라즈베리파이 + 마이크 + LED)는 모두 오픈소스 기반이며, 학부 수준에서 충분히 다룰 수 있는 기술 스택입니다. 또한 한이음에서 지원하는 실습장비비(최대 160만원)와 클라우드 환경으로 하드웨어 구매와 서버 운영이 가능합니다. 데이터 수집과 테스트도 팀원들의 실제 생활 공간에서 바로 진행할 수 있어, 외부 기관 연계 없이 자체적으로 개발-테스트-검증 사이클을 돌릴 수 있습니다.

| 기준 | 질문 | 이 프로젝트의 답 |
|------|------|-------------|
| 사회적 당위성 | 정말 도움이 필요한가? | ✅ 청각장애인의 가정 내 안전 사각지대 — SW만으로 해결 불가 |
| 시장 공백 | 이미 있는 서비스인가? | ✅ 한국에 AI 기반 환경음 감지 IoT 시스템 부재 |
| 구현 가능성 | 7개월 내 만들 수 있는가? | ✅ 오픈소스 기반 기술 + 한이음 장비 지원 + 자체 테스트 가능 |

이 프로젝트는 AI, 시스템 설계/통신, 프론트엔드(앱), HW(IoT) 4개 영역이 필요합니다. 5명의 팀원이 4개 영역을 빈틈 없이 커버할 수 있는지 확인한 결과, 핵심 기술 전부에 대응 가능한 인력이 있습니다.

### 1-1. 사회적 배경

- 한국 등록 청각장애인 수: 약 40만 명 (2023년 기준, 보건복지부 장애인 실태조사)
- 청각장애인은 가정 내 위험 상황(화재경보, 가스 누출 경보, 초인종, 아기 울음 등)을 인지하지 못해 안전 사각지대에 놓임
- 특히 취침 중이나 보청기를 제거한 상태에서는 위험 감지가 전혀 불가능
- 1인 가구 청각장애인의 경우 도움을 요청할 사람이 없어 위험이 가중됨
- 아이를 둔 청각장애인 부모의 경우, 아기 울음소리를 인지하지 못해 즉각적인 반응이 불가능한 실질적 육아 위험이 존재

### 1-2. 왜 "소프트웨어만"으로는 부족한가

| 상황 | 소프트웨어만 있을 때 | SW + HW 결합 시 |
|------|----------------|---------------|
| 취침 중 화재경보 | 스마트폰이 멀리 있으면 진동 못 느낌 | 침실 벽면 LED가 빨간색으로 점멸 + 베개 밑 진동 모터 작동 |
| 다른 방에서 초인종 | 스마트폰 마이크 범위 밖 → 감지 불가 | 해당 공간의 IoT 디바이스가 감지 → 모든 방 LED 알림 |
| 아기 울음 (육아 중) | 앱 알림에만 의존 → 폰 확인 안 하면 놓침 | 거실 LED가 노란색으로 즉시 변경, 시각적으로 바로 인지 |
| 외출 중 가족이 집에 있을 때 | 가족이 앱을 확인해야 상황 파악 | 보호자 앱에 실시간 푸시 + 감지 이력 자동 기록 |

### 1-3. 기존 솔루션의 한계

| 기존 솔루션 | 접근 방식 | 한계점 |
|-----------|---------|-------|
| **구글 Sound Notifications** | 스마트폰 앱 (SW only) | 단일 마이크, 다른 방 감지 불가, 한국 환경음 미특화 |
| **소리우산 (한국)** | 스마트밴드 연동 | 소리 크기(볼륨) 기반 → AI 종류 분류 아님 |
| **Sound Alert / Braci (해외)** | 스마트폰 앱 | 현재 서비스 사실상 중단 |
| **Earzz (영국)** | 전용 HW 디바이스 | 한국 미출시, 한국 환경음 미지원, 고가 (€499) |
| **iOS/Android 내장 기능** | OS 접근성 기능 | 스마트폰 범위 한정, 커스터마이징 제한 |

**공통 한계:** 대부분 소프트웨어 중심 접근으로, 청각장애인이 실제 가정에서 겪는 "소리를 다른 감각으로 전환해야 하는" 물리적 문제를 근본적으로 해결하지 못함

### 1-4. 프로젝트의 필요성 요약

1. **"기술의 가장 깊은 사각지대"에 있는 청각장애인**을 위한 프로젝트
2. 소프트웨어만으로는 해결 불가능한 문제를 **SW(AI 분류) + HW(IoT 디바이스)** 결합으로 해결
3. **한국 시장에 AI 기반 가정 환경음 분류 전용 IoT 시스템이 없음** — 시장 공백
4. 기존 솔루션은 **단일 공간, 단일 디바이스**에 한정 → 가정 전체를 커버하는 다중 IoT 구조 필요
5. 해외 서비스는 **한국 아파트 환경음(인터폰, 밥솥, 세탁기 등)**에 최적화되지 않음
6. 청각장애인의 **가정 내 안전권 보장 + 독립적 생활 지원**이라는 명확한 사회공헌 가치

### 3-2. 기술 스택

> ℹ️ 아래는 **프로젝트 구현에 권장하는 기술 스택**입니다. 팀 보유 역량과는 별개로, 이 프로젝트를 7개월 내에 가장 효율적으로 완성하기 위한 구성입니다.

**[1단계] 핵심 기능 구현 (4~6월, 학점연계 기말발표)**

| 영역 | 기술 | 선정 이유 |
|------|------|---------|
| **HW** | Raspberry Pi 4/5, USB Microphone, WS2812B LED Strip, Vibration Motor | 교육용 IoT 플랫폼, 학습 곡선 낮음, 디바이스당 약 5~8만원 |
| **AI/ML** | Python, TensorFlow, YAMNet, TFLite | YAMNet은 사전학습된 오디오 분류 모델, TFLite로 라즈베리파이 온디바이스 추론 가능 |
| **디바이스 통신** | MQTT (Mosquitto 브로커) | IoT 경량 메시징 표준, 다중 디바이스 간 실시간 연동에 최적 |
| **알림 전달** | 텔레그램 봇 API 또는 FCM 직접 호출 | 라즈베리파이에서 Python으로 즉시 구현 가능, 앱 개발 없이 스마트폰 알림 가능 |
| **이력 조회** | React 웹 대시보드 + Firebase Realtime Database | 팀원 B의 React 역량 활용, 간단한 웹페이지로 감지 이력 조회 |
| **형상 관리** | GitHub | 코드 관리 + 팀 협업 (수업 요구사항) |

**[2단계] 고도화 (7~10월, 한이음 공모전)**

| 영역 | 기술 | 추가 이유 |
|------|------|---------|
| **모바일 앱** | Flutter (Dart) | 전용 앱으로 푸시 알림 + 이력 조회 + UX 향상, 공모전 완성도 ↑ |
| **Backend** | Java, Spring Boot | Firebase 의존 탈피, 자체 서버로 전환, API 고도화 |
| **DB** | MySQL 또는 MongoDB | 대용량 이력 저장 + 통계 분석 |
| **웹 대시보드** | React (고도화) | 1차 간단 버전에서 시간대별 통계, 디바이스 관리 등 기능 확장 |
| **Infra** | AWS EC2 (한이음 클라우드 지원) | 자체 서버 운영 시 |

**왜 1단계에서 Spring Boot를 빼는가:**

이 프로젝트의 핵심은 "소리를 듣고 → AI가 분류하고 → 알려주는 것"입니다. 이 전체 과정은 라즈베리파이 + Firebase만으로 완결됩니다. Spring Boot 서버를 1단계부터 구축하면 핵심 기능(AI 분류 정확도, LED 알림 안정성)에 투입할 시간이 줄어듭니다. 중간평가까지는 핵심에 집중하고, 여유가 생기면 2단계에서 서버를 추가하는 것이 7개월 일정에서 가장 안전한 전략입니다.

**전체 데이터 흐름 (1단계 기준):**

```
[가정 내 IoT 디바이스]                    [클라우드/외부]            [사용자]
                                
┌─────────────┐                   ┌──────────────┐     ┌─────────────────┐
│ Raspberry Pi │──MQTT Publish──→│ MQTT Broker  │     │ 스마트폰         │
│ + USB Mic    │                  │ (Mosquitto)  │     │ 텔레그램 알림 수신 │
│ + AI Model   │──텔레그램 API───→│              │     └─────────────────┘
│ + LED Strip  │                  │ Firebase DB  │     ┌─────────────────┐
│ + Vibration  │──Firebase 저장──→│ (감지 이력)   │←───│ React 웹 대시보드 │
└─────────────┘                  └──────────────┘     │ (이력 조회)       │
      ×N대                              ↑             └─────────────────┘
 (주방, 거실, 침실)          다른 RPi도 MQTT 구독하여
                            LED/진동 연동
```

### 3-3. AI 모델 파이프라인

```
[마이크 입력] → [오디오 전처리 (16kHz, Mono)] → [Mel Spectrogram 변환]
     → [YAMNet Feature Extractor] → [Custom Classifier (Fine-tuned)]
     → [분류 결과 (소리 종류 + 신뢰도)] → [임계값 이상 시 알림 트리거]
```

**목표 정확도:**
- 긴급음(화재경보, 가스경보): 95% 이상
- 일상음(초인종, 아기울음 등): 85% 이상

**학습 상세:**
- Base Model: YAMNet (Google AudioSet, 521 클래스 사전학습)
- Fine-tuning: 상위 레이어를 한국 가정 환경음 데이터셋으로 재학습
- 배포: TFLite 변환 → 라즈베리파이 온디바이스 추론
- 데이터셋: 10종, 2000+ 샘플 (자체 수집)

## 5. 추진 일정

### 5-0. 일정 설계의 핵심 원칙

이 프로젝트는 **두 개의 마감**이 있습니다.

| 마감 | 시점 | 평가 주체 | 핵심 요구 |
|------|------|---------|---------|
| **1차: 학점연계 기말발표** | 6월 중순 | 교수님 | 작동하는 프로토타입 + 시연 + 결과보고서 |
| **2차: 한이음 공모전** | 10월 | 공모전 심사위원 | 완성도 높은 시스템 + 시연 영상 + 논문 |

6월 중순까지 교수님에게 보여줄 결과물이 부실하면 학점에 직접 타격을 받습니다. 따라서 **6월 중순을 "1차 완성"으로 설정하고, 이 시점에 독립적으로 시연 가능한 결과물**을 반드시 확보해야 합니다. 10월 공모전은 이를 기반으로 확장·고도화하는 2차 과정입니다.

---

### 5-1. 1차 과정 (4월 ~ 6월 중순) — 학점연계 기말발표

> 🎯 **1차 목표: "단일 디바이스에서 5~6종 한국 가정 환경음을 AI로 분류하고, LED 시각 알림 + 스마트폰 푸시 알림이 작동하는 프로토타입"**

| 월 | 마일스톤 | 완료 기준 |
|----|---------|---------|
| **4월** | 환경 세팅 + 첫 엔드투엔드 작동 | RPi에서 소리를 내면 YAMNet 기본 모델이 분류하고 LED 색상이 바뀌는 것을 팀 전원이 확인 |
| **5월** | 데이터셋 구축 + 모델 파인튜닝 + 앱 연동 | 한국 환경음 데이터셋으로 파인튜닝한 모델이 RPi에서 5~6종을 분류하고, 앱에 푸시 알림이 도달 |
| **6월 초~중** | 통합 안정화 + 기말발표 준비 | RPi → LED 알림 + 앱 푸시가 안정적으로 작동, 시연 영상 촬영, 결과보고서 제출 |

**4월의 핵심은 "일단 돌아가는 걸 보는 것"입니다.** 정확도가 낮아도, 코드가 지저분해도 상관없습니다. "소리 → AI → LED"가 하나로 연결되는 경험을 팀 전체가 빨리 공유해야 이후 방향이 잡힙니다.

**1차 완료 시 확보되는 것:**

- 라즈베리파이 1대에서 5~6종 한국 환경음 AI 분류 (목표 정확도: 위험음 90%+, 일상음 80%+)
- LED 시각 알림 (소리 종류별 색상 구분)
- 텔레그램 봇을 통한 스마트폰 알림
- React 웹 대시보드에서 감지 이력 조회
- 한국 가정 환경음 데이터셋 v1 (10종, 2000+ 샘플)
- 결과보고서 + 시연 영상 (1분)

## 6. 기대효과 및 활용방안

### 6-1. 기대효과

**기술적 기대효과:**

**사회적 기대효과:**
- 청각장애인의 가정 내 안전 사각지대 해소
- 화재 등 긴급 상황에서 골든타임 확보 지원
- 보호자의 원격 모니터링을 통한 돌봄 부담 경감
- 청각장애인 1인 가구의 독립적 생활 지원

**근거 데이터:**

> 현대 주택 화재 골든타임은 **1~2분(최대 3분)**이다. 과거 대비 합성 소재 가구로 불이 8배 빠르게 번지며, 1분만 인지가 지연되어도 플래시오버에 노출되어 사망률이 급증한다.
> — 출처: NFPA, 2022, 'Reporter's Guide: The consequences of fire'

> 갤러뎃 대학교 연구에 따르면, 소리 알람의 청각장애인 기상률은 33~57%에 불과하나, **진동경보기(Bed shaker) 사용 시 기상률이 90~100%**로 급상승한다.
> — 출처: Gallaudet University / Combustion Science & Engineering, 2006

**구체적 시나리오:**
> 새벽 2시 화재 발생 → RPi가 화재경보음 감지 → 침실 LED 빨간색 점멸 + 진동 모터 작동 → 청각장애인 30초 내 인지 → 골든타임 내 대피 가능

## 10. 차별화 포인트 정리 (발표 시 강조)

### vs 구글 Sound Notifications
- 구글: 스마트폰 마이크 1개 → 폰이 있는 방만 감지
- **우리: 다중 IoT 디바이스 → 가정 전체 커버**

### vs 해외 서비스 (Earzz 등)
- 해외: 서양 환경음 기준 학습, 한국 미출시
- **우리: 한국 아파트 특화 환경음 (인터폰, 밥솥, 세탁기 등) AI 학습**

### vs 소리우산 (한국)
- 소리우산: 소리 크기(볼륨) 감지 → 종류 구분 불가
- **우리: AI가 소리 종류를 자동 분류 → 상황별 맞춤 알림**

### vs 스마트폰 전용 앱
- 앱: 취침 시 폰 멀리 두면 무용
- **우리: 전용 LED 디바이스 → 취침 중에도 시각 알림 가능**

## 📂 [수행계획서/최종보고서용] 정책적 배경 — SW 편중 문제

> ⚠️ **이 섹션은 1주차 발표에는 포함하지 않습니다.**
> 2주차 계획 발표, 한이음 수행계획서, 또는 최종 보고서의 "추진 배경 및 필요성" 항목에서 활용하세요.

### 핵심 논지

현재 한국의 디지털 포용 정책은 "스마트폰을 쓸 수 있는가", "웹사이트에 접근할 수 있는가"를 측정하고 개선하는 **소프트웨어 접근성**에 집중되어 있습니다. 장애인의 디지털 접근 수준은 96.5%까지 올라갔지만, 이것은 "화면을 보고 터치하는" 환경에서의 접근성일 뿐입니다. 청각장애인이 가정에서 화재경보를 인지할 수 있는가, 초인종 소리를 들을 수 있는가 — 이런 **물리적 환경에서의 정보 접근**은 정책의 사각지대에 놓여 있습니다.

### 근거 자료

**1. 디지털 정보격차 실태조사 (2024, 과학기술정보통신부)**

- 2024년 취약계층의 디지털 정보화 수준은 평균 77.5%로 전년 대비 0.6%p 개선
- **장애인의 디지털 접근 수준은 96.5%** — 스마트폰/컴퓨터 보유 및 기본 이용은 거의 해결됨
- 그러나 이 조사가 측정하는 항목은 전부 소프트웨어적 접근(웹사이트, 앱, 무인단말기)
- **"가정 내 환경음 인지 가능 여부" 같은 물리적 환경에서의 정보 접근은 측정 항목에 포함되어 있지 않음**
- 출처: 과학기술정보통신부 보도자료 (2025.03.27), 더인디고 (2025.03.27)

**2. 장애인차별금지법 이행 실태조사 (2024, 보건복지부)**

- 조사 대상: 무인정보단말기와 응용 소프트웨어의 설치·운영 실태 → SW/키오스크 중심
- 청각장애인의 38.0%가 "편의 기능 미비·부족으로 무인정보단말기 이용이 어렵다"고 응답
- 장애인차별금지법의 정당한 편의 제공 의무도 무인단말기·앱·웹 등 **소프트웨어 환경**에 집중
- **가정 내 안전 환경(화재감지, 초인종 알림 등)에 대한 접근성 조사는 미실시**
- 출처: 보건복지부 보도자료 (2024)

**3. 등록장애인 현황 (2024, 보건복지부)**

- 2024년 말 기준 등록장애인: 2,631,356명 (전체 인구의 5.1%)
- **청각장애: 16.8%로 두 번째로 많은 비율** (약 44만 명)
- 2024년 신규 등록 장애인 중 **청각장애가 31.7%로 1위** — 계속 증가 추세
- 65세 이상 장애인 비율: 55.3% → 고령 청각장애인은 디지털 기기 활용 자체가 더 어려움
- 출처: 보건복지부 보도자료 (2025.04.18)

### 수행계획서에 활용 시 권장 문구 (예시)

> *"과학기술정보통신부의 2024년 디지털 정보격차 실태조사에 따르면, 장애인의 디지털 접근 수준은 96.5%에 달하며 소프트웨어 기반의 정보 접근성은 크게 개선되었다. 그러나 이러한 조사와 정책은 웹사이트·앱·무인단말기 등 소프트웨어 환경에서의 접근성에 집중되어 있으며, 청각장애인이 가정 내에서 화재경보·초인종 등 환경음을 인지할 수 있는지에 대한 물리적 환경의 정보 접근은 정책적 사각지대에 놓여 있다. 본 프로젝트는 이러한 'SW 편중' 문제를 인식하고, AI 소프트웨어와 IoT 하드웨어의 결합을 통해 청각장애인의 가정 내 안전 사각지대를 해소하고자 한다."*

### 논리 구조 요약

```
[현재 상태]
정부 디지털 포용 정책 → SW 접근성 집중 (웹, 앱, 키오스크)
→ 장애인 디지털 접근 수준 96.5% 달성 ✅

[사각지대]
그러나 "소리를 통한 정보 인지"는 SW로 해결 불가
→ 화재경보, 초인종, 아기 울음 = 청각 채널에 의존
→ 청각장애인에게는 SW 접근성 96.5%가 무의미한 영역 ❌

[우리의 접근]
SW(AI 오디오 분류) + HW(IoT 디바이스, LED, 진동)
→ 소리를 시각·촉각으로 전환하여 물리적 사각지대 해소 🎯
```
